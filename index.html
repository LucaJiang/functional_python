Functional Python

Practical Paradigms for Readable and Robust Code

Wenxin Jiang · PyCon HK

Oct. 11, 2025

While Python's object-oriented paradigm dominates developer discussions, its functional capabilities offer powerful tools for modern development. This talk explores how integrating functional programming (FP) concepts can enhance code readability and maintainability in Python projects. We will discover how to transform imperative logic flow into composable functions and implement declarative error-handling using intuitive railway patterns. This talk aims to provide an optional toolbox for Python developers to improve code quality. Functional programming (FP) offers powerful tools and is increasingly adopted across modern languages, including Python. Since Python is not a purely functional language, rather than advocating for a complete shift to FP, this talk will demonstrate how to effectively integrate functional techniques, such as pure functions, higher-order functions and Monads, into your Python codebase. By converting imperative procedure-style logic into composable functions, developers can focus more on the "what" (logic) rather than the "how" (implementation details). As for error handling, we will explore how to organize code using a fun and easy-to-understand railway analogy. By the end of this talk, you'll understand how to apply basic FP concepts in Python, which improve code's readability, maintainability, and robustness.

What is FP?

Core: Pure Functions

  • Functions with no side effects
  • Avoid global state
  • Same result for the same input
  • One function, one task
n a functional program, input flows through a set of functions. Each function operates on its input and produces some output. Functional style discourages functions with side effects that modify internal state or make other changes that aren’t visible in the function’s return value. Functions that have no side effects at all are called purely functional. Avoiding side effects means not using data structures that get updated as a program runs; every function’s output must only depend on its input. Functional programming is a style of coding where the focus is on writing functions that don’t alter the state of the program. Key principles of FP include pure functions—functions that always give the same output for the same input, and immutability, where data cannot be changed once created. These principles make the code easier to test, reason about, and scale.

From the function perspective, we emphasize "no side effects" - functions should only depend on their inputs and not modify external state. These two concepts are two sides of the same coin, working together to make code more reliable and easier to reason about.

Immutability: The Problem

❌ Mutable Approach

scores = [85, 74, 60] # Shared mutable state

def add_bonus(score_list):
    for i in range(len(score_list)):
        score_list[i] += 10  # Modifies original!
    return score_list

original_scores = scores
bonus_scores = add_bonus(scores)

print(original_scores) # Output: [85, 74, 60]  😱 Changed!

⚠️ Unexpected mutation causes bugs!

This example demonstrates a common bug with mutable data. The function modifies the original list directly. This causes unexpected behavior because both original_scores and bonus_scores point to the same mutated list. This is especially dangerous in larger programs where you might not realize data is being shared and modified.

Immutability: The Solution

✅ Immutable Approach

scores = [85, 74, 60] # Immutable approach

def add_bonus(score_list):
    # Create new list, don't modify original
    return [score + 10 for score in score_list]

original_scores = scores
bonus_scores = add_bonus(scores)

print(original_scores)
# Output: [85, 74, 60]  ✓ Unchanged!

✓ Original data is safe!

The immutable approach creates a new list instead of modifying the original. This ensures the original data remains unchanged, making the code predictable and preventing subtle bugs. This is a key principle of functional programming - always create new data structures instead of modifying existing ones.

Why FP?

  • Readability: Focus on "what" rather than "how"
  • Modularity: Small, isolated functions
  • Robustness/Testability: Easier to test
Functional programming offers several advantages that can significantly enhance the quality of your code. First, it improves readability by allowing you to focus on "what" the code is doing rather than "how" it is implemented. This is achieved through the use of pure functions and higher-order functions, which make the code more declarative and easier to understand. These are some advantages of functional programming: • Absence of side effects can make your programs more robust • Programs tend to be more modular and come typically in smaller building blocks • Better testable - a call with same parameters always returns same result due to no global state (except when intended such as random number generation or IO) • Focus on algorithms • Conceptional fit with parallel / concurrent programming • Live updates - Install new release while running (Erlang)

Currying & Higher-Order Functions

Source: Currying takes your functions to a whole new level of flexibility and power!

Currying is a technique in functional programming where a function that takes multiple arguments is transformed into a sequence of functions, each taking a single argument. In the uncurrying example, we pass all arguments at once. In the currying example, we pass one argument at a time, with each function call returning another function until all arguments are provided. This allows for partial application and more flexible function composition.

Currying: Why Important?

Math to Code: Function Composition

$h(g(f(x))) = (h \circ g \circ f)(x)$

❌ Nested (Hard to read)

result = h(g(f(df), arg1=a), arg2=b)

✅ Pipeline (Clear flow)

result = df.pipe(f).pipe(g, arg1=a).pipe(h, arg2=b)
This slide demonstrates why currying is important. Mathematically, it's related to function composition. In programming, currying allows us to transform nested function calls into a clean pipeline. The nested version requires reading from inside out, while the pipeline version creates a clear, left-to-right data flow that's much easier to read and maintain.

Railway Analogy for Error Handling

Source: Railway Oriented Programming

Railway Analogy for Error Handling

Source: Railway Oriented Programming

This slide illustrates the railway pattern for error handling. The idea is to represent the flow of data through a series of functions as two parallel tracks: one for successful operations and another for errors. Each function takes an input and returns either a success or an error. If a function encounters an error, it switches to the error track, and subsequent functions are skipped. This approach allows for clear and concise error handling without deeply nested try-except blocks, making the code easier to read and maintain.

Other Interesting Concepts in FP

  • Lazy Evaluation: Delay computation until necessary, improving performance.
  • Recursion: Use recursive functions for problems like tree traversal.
  • Monads: Handle side effects (like state, I/O) in a controlled manner.
  • Pattern Matching: Simplify conditional logic based on data structure.
